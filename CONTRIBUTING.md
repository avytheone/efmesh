# Как контрибьютить

Спасибо, что решили помочь! efmesh — молодой проект, правила простые.

## Окружение

Нужен только [Bun](https://bun.sh) ≥ 1.3 (рантайм, тесты, пакетный менеджер):

```sh
bun install     # зависимости
bun test        # все тесты (~4 с)
bun run check   # tsc --noEmit
```

Postgres-тесты поднимают одноразовый кластер через `initdb` в tmp и
автоматически пропускаются, если бинарников Postgres нет в PATH —
для полного прогона поставьте `postgresql` локально.

## Устройство репозитория

```
SPEC.md            — архитектурная спецификация: читать перед крупной правкой
src/core/          — модели, DAG, SQL-фрагменты, интервальная арифметика
src/engine/        — адаптеры движков (DuckDB, Postgres)
src/state/         — state store (bun:sqlite, Postgres)
src/plan/          — fingerprint, планировщик, executor, аудиты, janitor, lock
src/cli.ts         — CLI (тонкая обёртка над библиотекой)
src/testing/       — testModel для юнит-тестов моделей
test/              — bun test; один файл на область
examples/hospital/ — живой пример, на нём гоняется dogfood
```

## Принципы кода

- **Effect везде.** Ошибки — типизированные (`Data.TaggedError`), никаких
  `throw` в бизнес-логике; ресурсы — `Layer`/`Scope`; форма данных —
  `Schema`. Целевая версия — Effect v4 (beta): держимся стабильного
  подмножества API.
- **SPEC.md — источник правды.** Поведенческие изменения сначала
  отражаются в спеке (или обсуждаются issue), потом в коде. Реализованное
  помечается в спеке уточнениями «*Реализация Fn: …*».
- **Тест на каждое поведение.** Правка без падающего до неё теста —
  повод спросить, что именно она чинит.
- **Dogfood.** После заметной правки прогоните живой цикл на примере:
  `cd examples/hospital && bun ../../src/bin.ts apply dev --yes && bun ../../src/bin.ts run dev`.

## Коммиты и PR

- Атомарные коммиты в стиле [Conventional Commits](https://www.conventionalcommits.org/ru/):
  `feat(plan): …`, `fix(state): …`, `docs(spec): …` — по одному
  законченному изменению на коммит.
- В теле коммита — «почему», а не «что» (что видно из diff). Найденные
  грабли документируйте — они экономят часы следующему.
- Перед PR: `bun run check && bun test` зелёные. Проверяйте код выхода
  честно — pipe в `tail` его глотает.
